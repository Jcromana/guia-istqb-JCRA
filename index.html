Capítulo 1: Fundamentos del Testing

Este capítulo introduce los conceptos básicos, la terminología y los principios que forman la base del testing de software profesional. Entender estos fundamentos es esencial para aplicar eficazmente las técnicas y procesos en cualquier proyecto.

¿Qué es el Testing y por qué es necesario?

El testing de software es un proceso para evaluar un producto y determinar si cumple con los requisitos especificados, identificando defectos para mejorar la calidad. Es una parte crucial del aseguramiento de la calidad (QA).

Es necesario para mitigar riesgos, mejorar la calidad del producto y cumplir con requisitos contractuales o legales, asegurando que el software es fiable y satisface las necesidades del usuario.

Ejemplo:

  • Error: Un desarrollador escribe `>` en lugar de `>=` en una condición para validar la edad.
  • Defecto (Bug): El código ahora rechaza incorrectamente a los usuarios que tienen exactamente 18 años.
  • Fallo: Un usuario de 18 años intenta registrarse y el sistema le muestra un mensaje de "Edad no permitida".

Siete Principios del Testing

1. El testing demuestra la presencia de defectos, no su ausencia.

Ejemplo: Aunque probemos 1000 inicios de sesión exitosos, no podemos garantizar que el 1001 no fallará.

2. Es imposible realizar pruebas exhaustivas.

Ejemplo: Probar un campo de texto con cada carácter Unicode posible es inviable; en su lugar, usamos técnicas para seleccionar los más relevantes.

3. Las pruebas tempranas ahorran tiempo y dinero.

Ejemplo: Encontrar un error en el documento de requisitos cuesta minutos; encontrarlo en producción puede costar miles de euros.

4. Los defectos se agrupan (Principio de Pareto).

Ejemplo: Es común que el 80% de los defectos se encuentren en el 20% de los módulos, como el de procesamiento de pagos.

5. La paradoja del pesticida.

Ejemplo: Si siempre probamos el login con "user/pass", dejaremos de encontrar nuevos fallos. Hay que variar los datos.

6. El testing depende del contexto.

Ejemplo: Un videojuego requiere pruebas de jugabilidad y rendimiento gráfico, mientras que una app bancaria exige máxima seguridad.

7. La falacia de la ausencia de errores.

Ejemplo: Un software sin fallos pero que es difícil de usar y no resuelve el problema del cliente, es inútil.

Proceso de Testing

Planificación

Definir qué se va a probar, cómo, quién lo hará y cuándo. Se crea el Plan de Pruebas.

Monitorización y Control

Seguimiento del progreso con métricas. Si las pruebas se retrasan, el control implica tomar acciones.

Análisis

Estudiar los requisitos para identificar las condiciones de prueba. ¿Qué podemos probar?

Diseño

Crear casos de prueba de alto nivel. ¿Cómo vamos a probarlo?

Implementación

Escribir los pasos detallados de los casos de prueba y preparar el entorno y los datos.

Ejecución

Ejecutar los casos de prueba, comparar resultados y reportar discrepancias.

Finalización

Recopilar resultados, crear informes finales y archivar los artefactos de prueba.

Capítulo 2: Testing a lo largo del Ciclo de Vida del Software

El testing no es una fase aislada, sino una actividad integrada en todo el ciclo de vida del desarrollo de software. Este capítulo explora cómo las actividades de testing se alinean con diferentes modelos de desarrollo y los distintos niveles y tipos de prueba que se aplican.

Niveles de Prueba

Pruebas de Componente (Unitarias): Se centran en probar componentes individuales de forma aislada.

Ejemplo: Probar una función `calcularIVA()` para asegurar que devuelve el valor correcto para diferentes entradas.

Pruebas de Integración: Verifican la interacción entre componentes o sistemas integrados.

Ejemplo: Probar que el módulo de "carrito de la compra" se comunica correctamente con el módulo de "pasarela de pago".

Pruebas de Sistema: Evalúan el sistema completo para verificar que cumple con los requisitos especificados.

Ejemplo: Probar el flujo completo de una compra online, desde añadir un producto al carrito hasta recibir el email de confirmación.

Pruebas de Aceptación (UAT): Realizadas por los usuarios para validar que el sistema cumple con sus necesidades.

Ejemplo: Un cliente final prueba la nueva funcionalidad de "reservar cita" para confirmar que es intuitiva y funciona como esperaba.

Tipos de Prueba

Pruebas Funcionales: Verifican qué hace el sistema.

Ejemplo: Probar que al hacer clic en "Exportar a PDF", se genera un archivo PDF válido.

Pruebas No Funcionales: Verifican cómo funciona el sistema.

Ejemplo: Probar que la página de inicio carga en menos de 3 segundos con 100 usuarios concurrentes (rendimiento).

Pruebas de Confirmación (Re-testing): Confirman que un defecto ha sido corregido.

Ejemplo: Un bug causaba un error al pagar con American Express. Tras el arreglo, se vuelve a probar el pago con esa tarjeta.

Pruebas de Regresión: Aseguran que los cambios no han introducido nuevos defectos en funcionalidades existentes.

Ejemplo: Tras arreglar el bug de American Express, se prueba que el pago con Visa y Mastercard siga funcionando correctamente.

Testing en el contexto de DevOps

DevOps integra el desarrollo y las operaciones para acortar el ciclo de vida del desarrollo. El testing es una actividad continua y automatizada, no una fase.

  • Integración Continua (CI): Las pruebas automatizadas se ejecutan cada vez que se integra código nuevo.
  • Entrega/Despliegue Continuo (CD): El software que pasa las pruebas se puede desplegar automáticamente.
  • Shift-Left: El testing se "desplaza a la izquierda", comenzando lo antes posible en el ciclo de vida para una retroalimentación más rápida.
Ejemplo: Un desarrollador sube un cambio. El servidor de CI (ej. Jenkins) compila el código, ejecuta pruebas unitarias y de integración. Si todo es correcto, lo despliega en un entorno de pruebas automáticamente para las pruebas de sistema.

Capítulo 3: Pruebas Estáticas

A diferencia de las pruebas dinámicas que ejecutan el código, las pruebas estáticas examinan los productos de trabajo (documentos, código) para encontrar defectos. Son una forma muy rentable de mejorar la calidad desde las primeras fases del proyecto.

Fundamentos y Proceso de Revisión

Las pruebas estáticas encuentran defectos directamente en los productos de trabajo, sin necesidad de ejecutar el software. Su principal beneficio es la detección temprana de defectos, lo que reduce significativamente el costo de corrección.

Ejemplo: Durante la revisión de un documento de requisitos, un tester detecta una ambigüedad: "El sistema debe ser rápido". Esto es un defecto en la documentación que se corrige antes de escribir una sola línea de código.

Tipos de Revisión

Revisión Informal

Ejemplo: Un desarrollador le pide a un compañero que eche un vistazo rápido a su código antes de subirlo.

Walkthrough

Ejemplo: Un analista de negocio presenta los nuevos mockups de la interfaz a desarrolladores y testers para recoger feedback inicial.

Revisión Técnica

Ejemplo: Un arquitecto de software lidera una sesión para revisar el diseño técnico de un nuevo microservicio con otros desarrolladores senior.

Inspección

Ejemplo: Se realiza una inspección formal del plan de pruebas de un sistema crítico, con métricas de defectos encontrados por página y seguimiento formal.

Capítulo 4: Análisis y Diseño de Pruebas

Este capítulo presenta las técnicas sistemáticas que utilizan los testers para diseñar casos de prueba eficaces. Estas técnicas ayudan a seleccionar un subconjunto óptimo de pruebas para maximizar la detección de defectos con un esfuerzo limitado.

Partición de Equivalencia y Análisis de Valores Límite:

Un campo de edad acepta valores de 18 a 65.
• Particiones: `(< 18)`, `(18-65)`, `(> 65)`. Probaríamos con 17, 30, 66.
• Valores Límite: Probaríamos con 17, 18, 19 y 64, 65, 66.

Tabla de Decisión:

Login: Condiciones (Usuario válido, Contraseña válida). Acciones (Acceso permitido, Mensaje de error). La tabla tendría 4 reglas para cubrir todas las combinaciones (V/V, V/F, F/V, F/F).

Transición de Estados:

Un pedido online: `Pendiente` -> (al pagar) -> `Procesando` -> (al enviar) -> `Enviado`. Se prueban estas transiciones.

Cobertura de Sentencias y Decisión:

IF edad >= 18 THEN
  imprimir("Permitido")
ELSE
  imprimir("Denegado")
ENDIF
• Cobertura de Sentencias (100%): Probar con edad = 20 (ejecuta la línea 2). Probar con edad = 15 (ejecuta la línea 4). Se necesitan 2 casos de prueba.
• Cobertura de Decisión (100%): Probar con edad = 20 (la condición es Verdadera). Probar con edad = 15 (la condición es Falsa). Se necesitan 2 casos de prueba.

Pruebas Exploratorias:

Un tester recibe una nueva app de e-commerce. En lugar de seguir un script, explora libremente: ¿Qué pasa si añado 1000 productos al carrito? ¿Y si uso emojis en el campo de dirección? Anota sus hallazgos y reporta bugs.

Adivinación de Errores:

Un tester experimentado sabe que las divisiones por cero suelen causar problemas, así que prueba a introducir un '0' en todos los campos numéricos que puedan ser divisores.

Capítulo 5: Gestión de las Actividades de Prueba

La gestión de pruebas abarca la planificación, estimación, monitorización y control de todas las actividades de testing. Este capítulo cubre cómo organizar el trabajo, gestionar riesgos y manejar los defectos de manera efectiva.

Planificación y Estimación

Se documenta en el Plan de Pruebas, que define el alcance, la estrategia y los recursos. Incluye criterios de entrada y de salida.

• Criterio de Entrada: "El entorno de pruebas debe estar instalado y estable."
• Criterio de Salida: "El 95% de los casos de prueba críticos deben haberse ejecutado y aprobado."

Técnicas de Estimación

  • Basadas en métricas: Usar la duración media de pruebas de un proyecto similar.
  • Basadas en la opinión de expertos: En Planning Poker, el equipo vota con cartas para estimar el esfuerzo de una tarea.

Monitorización y Control

Se recopilan métricas para seguir el progreso (monitorización) y se toman acciones correctivas si hay desviaciones del plan (control).

• Métrica: El número de defectos abiertos aumenta en lugar de disminuir.
• Acción de Control: Se dedica un día entero a estabilizar el software y a realizar pruebas de regresión en lugar de probar nuevas funcionalidades.

Gestión de Riesgos

Consiste en identificar, analizar y mitigar los riesgos del producto para priorizar el esfuerzo de testing en las áreas más críticas.

• Riesgo: "El rendimiento del sistema de informes puede ser lento con grandes volúmenes de datos."
• Mitigación: Se planifican pruebas de carga específicas para el módulo de informes.

Gestión de Defectos

Un informe de defectos claro es crucial. El ciclo de vida de un defecto describe su paso desde que se reporta hasta que se cierra.

Título de Defecto: "[Cálculo de Envío] El coste es incorrecto para pedidos a las Islas Canarias."

Capítulo 6: Herramientas de Soporte al Testing

Las herramientas pueden hacer que el proceso de testing sea más eficiente y efectivo. Este capítulo clasifica los diferentes tipos de herramientas y discute los beneficios y riesgos de la automatización de pruebas.

Clasificación de Herramientas

  • Herramientas de gestión: Para gestión de pruebas y defectos (ej. Jira, TestRail, Zephyr).
  • Herramientas para pruebas estáticas: Analizadores de código estático (ej. SonarQube, ESLint).
  • Herramientas de ejecución y registro: Para automatización de pruebas (ej. Selenium, Cypress, Playwright).
  • Herramientas de rendimiento: Para pruebas de carga y estrés (ej. JMeter, Gatling, k6).

Automatización de Pruebas

Beneficios

  • Ejecución rápida y repetible
  • Reducción de errores humanos
  • Libera a los testers para tareas complejas

Riesgos

  • Expectativas poco realistas
  • Alto costo inicial y de mantenimiento
  • Scripts frágiles ante cambios
Buen candidato para automatizar: Una suite de pruebas de regresión que verifica las 20 funcionalidades más críticas del sistema después de cada cambio.
Mal candidato para automatizar: Una prueba de usabilidad para ver cómo reacciona un nuevo usuario a un rediseño de la interfaz.
