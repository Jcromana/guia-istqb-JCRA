Capítulo 1: Fundamentos del Testing

Este capítulo introduce los conceptos básicos, la terminología y los principios que forman la base del testing de software profesional. Entender estos fundamentos es esencial para aplicar eficazmente las técnicas y procesos en cualquier proyecto.

¿Qué es el Testing y por qué es necesario?

El testing de software es un proceso para evaluar un producto y determinar si cumple con los requisitos especificados, identificando defectos para mejorar la calidad. Es una parte crucial del aseguramiento de la calidad (QA).

Es necesario para mitigar riesgos, mejorar la calidad del producto y cumplir con requisitos contractuales o legales, asegurando que el software es fiable y satisface las necesidades del usuario.

Error: Una acción humana que produce un resultado incorrecto.

Defecto (Bug): El resultado de un error en el código o documento.

Fallo: La manifestación visible de un defecto cuando el software no se comporta como se espera.

Siete Principios del Testing

1. El testing demuestra la presencia de defectos, no su ausencia.
2. Es imposible realizar pruebas exhaustivas.
3. Las pruebas tempranas ahorran tiempo y dinero.
4. Los defectos se agrupan (Principio de Pareto).
5. La paradoja del pesticida.
6. El testing depende del contexto.
7. La falacia de la ausencia de errores.

Proceso de Testing

Planificación

Definir estrategia, objetivos y recursos.

Monitorización y Control

Comparar progreso con el plan y corregir desviaciones.

Análisis

Revisar la base de pruebas para identificar qué probar.

Diseño

Especificar casos de prueba de alto nivel.

Implementación

Crear casos de prueba detallados y preparar el entorno.

Ejecución

Ejecutar pruebas y registrar resultados.

Finalización

Recopilar resultados y elaborar informes finales.

Capítulo 2: Testing a lo largo del Ciclo de Vida del Software

El testing no es una fase aislada, sino una actividad integrada en todo el ciclo de vida del desarrollo de software. Este capítulo explora cómo las actividades de testing se alinean con diferentes modelos de desarrollo y los distintos niveles y tipos de prueba que se aplican.

Niveles de Prueba

Pruebas de Componente (Unitarias): Se centran en probar componentes individuales de forma aislada.

Pruebas de Integración: Verifican la interacción entre componentes o sistemas integrados.

Pruebas de Sistema: Evalúan el sistema completo para verificar que cumple con los requisitos especificados.

Pruebas de Aceptación (UAT): Realizadas por los usuarios para validar que el sistema cumple con sus necesidades.

Tipos de Prueba

Pruebas Funcionales: Verifican qué hace el sistema (ej. iniciar sesión).

Pruebas No Funcionales: Verifican cómo funciona el sistema (rendimiento, usabilidad, seguridad).

Pruebas de Confirmación (Re-testing): Confirman que un defecto ha sido corregido.

Pruebas de Regresión: Aseguran que los cambios no han introducido nuevos defectos en funcionalidades existentes.

Testing en el contexto de DevOps

DevOps integra el desarrollo y las operaciones para acortar el ciclo de vida del desarrollo. El testing es una actividad continua y automatizada, no una fase.

  • Integración Continua (CI): Las pruebas automatizadas se ejecutan cada vez que se integra código nuevo.
  • Entrega/Despliegue Continuo (CD): El software que pasa las pruebas se puede desplegar automáticamente.
  • Shift-Left: El testing se "desplaza a la izquierda", comenzando lo antes posible en el ciclo de vida para una retroalimentación más rápida.

Capítulo 3: Pruebas Estáticas

A diferencia de las pruebas dinámicas que ejecutan el código, las pruebas estáticas examinan los productos de trabajo (documentos, código) para encontrar defectos. Son una forma muy rentable de mejorar la calidad desde las primeras fases del proyecto.

Fundamentos y Proceso de Revisión

Las pruebas estáticas encuentran defectos directamente en los productos de trabajo, sin necesidad de ejecutar el software. Su principal beneficio es la detección temprana de defectos, lo que reduce significativamente el costo de corrección.

Un proceso de revisión formal incluye roles como Autor, Moderador y Revisor, y sigue fases como planificación, preparación individual y seguimiento.

Tipos de Revisión

Revisión Informal

Simple revisión por un compañero, sin proceso formal.

Walkthrough

El autor presenta el documento para obtener feedback.

Revisión Técnica

Proceso formal dirigido por un moderador para que expertos busquen defectos.

Inspección

El tipo de revisión más formal y riguroso, con métricas definidas.

Capítulo 4: Análisis y Diseño de Pruebas

Este capítulo presenta las técnicas sistemáticas que utilizan los testers para diseñar casos de prueba eficaces. Estas técnicas ayudan a seleccionar un subconjunto óptimo de pruebas para maximizar la detección de defectos con un esfuerzo limitado.

Partición de Equivalencia: Divide los datos de entrada en grupos equivalentes. Se prueba un valor de cada grupo.

Análisis de Valores Límite: Prueba los bordes o límites de las particiones, donde suelen ocurrir errores.

Tabla de Decisión: Para probar lógica de negocio compleja con múltiples condiciones.

Transición de Estados: Modela y prueba los cambios de estado de un sistema.

Pruebas de Casos de Uso: Verifica los flujos de interacción entre el usuario y el sistema.

Cobertura de Sentencias: Diseña pruebas para ejecutar cada línea de código al menos una vez.

Cobertura de Decisión: Diseña pruebas para que cada decisión en el código se evalúe como verdadera y falsa.

Pruebas Exploratorias: El tester diseña y ejecuta pruebas simultáneamente, aprendiendo sobre el sistema.

Adivinación de Errores: El tester usa su experiencia para "adivinar" dónde es probable que haya errores.

Capítulo 5: Gestión de las Actividades de Prueba

La gestión de pruebas abarca la planificación, estimación, monitorización y control de todas las actividades de testing. Este capítulo cubre cómo organizar el trabajo, gestionar riesgos y manejar los defectos de manera efectiva.

Planificación y Estimación

Se documenta en el Plan de Pruebas, que define el alcance, la estrategia y los recursos. Incluye criterios de entrada (cuándo empezar) y criterios de salida (cuándo terminar).

Técnicas de Estimación

  • Basadas en métricas: Usan datos de proyectos anteriores.
  • Basadas en la opinión de expertos: Como Planning Poker o Delphi.

Monitorización y Control

Se recopilan métricas para seguir el progreso (monitorización) y se toman acciones correctivas si hay desviaciones del plan (control).

Gestión de Riesgos

Consiste en identificar, analizar y mitigar los riesgos del producto para priorizar el esfuerzo de testing en las áreas más críticas.

Gestión de Defectos

Un informe de defectos claro es crucial. El ciclo de vida de un defecto describe su paso desde que se reporta hasta que se cierra.

Capítulo 6: Herramientas de Soporte al Testing

Las herramientas pueden hacer que el proceso de testing sea más eficiente y efectivo. Este capítulo clasifica los diferentes tipos de herramientas y discute los beneficios y riesgos de la automatización de pruebas.

Clasificación de Herramientas

  • Herramientas de gestión: Para gestión de pruebas y defectos (ej. Jira, TestRail).
  • Herramientas para pruebas estáticas: Analizadores de código estático.
  • Herramientas de ejecución y registro: Para automatización de pruebas (ej. Selenium, Cypress).
  • Herramientas de rendimiento: Para pruebas de carga y estrés (ej. JMeter).

Automatización de Pruebas

Beneficios

  • Ejecución rápida y repetible
  • Reducción de errores humanos
  • Libera a los testers para tareas complejas

Riesgos

  • Expectativas poco realistas
  • Alto costo inicial y de mantenimiento
  • Scripts frágiles ante cambios
